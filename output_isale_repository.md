## Contrato de Repositorio para Ventas (ISaleRepository.cs)

### 1. Código Fuente Completo de 'ISaleRepository.cs'

\`\`\`csharp
using System.Collections.Generic;
using System.Threading;
using System.Threading.Tasks;
using UtmMarket.Core.Entities;
using UtmMarket.Core.Filters;

namespace UtmMarket.Core.Repositories;

/// <summary>
/// Defines the contract for data access operations related to the <see cref="Sale"/> aggregate root.
/// This interface is part of the domain layer and must not depend on persistence implementation details.
/// </summary>
public interface ISaleRepository
{
    /// <summary>
    /// Retrieves all sales asynchronously.
    /// Utilizes <see cref="IAsyncEnumerable{T}"/> for data streaming,
    /// optimizing memory usage for potentially large result sets.
    /// </summary>
    /// <param name="cancellationToken">A token to cancel the asynchronous operation.</param>
    /// <returns>An asynchronous enumeration of <see cref="Sale"/>.</returns>
    IAsyncEnumerable<Sale> GetAllAsync(CancellationToken cancellationToken = default);

    /// <summary>
    /// Retrieves a specific sale by its unique identifier asynchronously.
    /// </summary>
    /// <param name="saleId">The unique identifier of the sale.</param>
    /// <param name="cancellationToken">A token to cancel the asynchronous operation.</param>
    /// <returns>A <see cref="Task{TResult}"/> representing the asynchronous operation,
    /// containing the found <see cref="Sale"/> or <c>null</c> if it does not exist.</returns>
    Task<Sale?> GetByIdAsync(int saleId, CancellationToken cancellationToken = default);

    /// <summary>
    /// Finds sales that match the criteria specified in the filter asynchronously.
    /// Utilizes <see cref="IAsyncEnumerable{T}"/> for data streaming of the results.
    /// </summary>
    /// <param name="filter">A <see cref="SaleFilter"/> object containing the search criteria.</param>
    /// <param name="cancellationToken">A token to cancel the asynchronous operation.</param>
    /// <returns>An asynchronous enumeration of <see cref="Sale"/> that satisfy the filter.</returns>
    IAsyncEnumerable<Sale> FindAsync(SaleFilter filter, CancellationToken cancellationToken = default);

    /// <summary>
    /// Adds a new sale to the repository asynchronously.
    /// </summary>
    /// <param name="sale">The <see cref="Sale"/> object to add. The sale's ID may be 0
    /// if it is generated by the data source.</param>
    /// <param name="cancellationToken">A token to cancel the asynchronous operation.</param>
    /// <returns>A <see cref="Task{TResult}"/> representing the asynchronous operation,
    /// containing the <see cref="Sale"/> with any persistence-generated ID.</returns>
    Task<Sale> AddAsync(Sale sale, CancellationToken cancellationToken = default);

    /// <summary>
    /// Updates an existing sale in the repository asynchronously.
    /// </summary>
    /// <param name="sale">The <see cref="Sale"/> object with the updated data.</param>
    /// <param name="cancellationToken">A token to cancel the asynchronous operation.</param>
    /// <returns>A <see cref="Task{TResult}"/> representing the asynchronous operation,
    /// containing <c>true</c> if the update was successful; otherwise, <c>false</c>.</returns>
    Task<bool> UpdateAsync(Sale sale, CancellationToken cancellationToken = default);
}

### 3. Asesoría Arquitectónica: Impacto de Native AOT en la Implementación de 'ISaleRepository'

La elección de Native AOT (Ahead-Of-Time compilation) para el proyecto "utmmarket" tiene implicaciones significativas para la futura implementación de la interfaz `ISaleRepository`. Aunque la interfaz misma es una abstracción pura de dominio, las implementaciones concretas en la capa de infraestructura deberán diseñarse cuidadosamente para aprovechar y ser compatibles con AOT.

**Beneficios y Consideraciones Clave:**

1.  **Restricciones de Reflexión y Generación de Código en Tiempo de Ejecución:**
    *   **Impacto:** Native AOT precompila todo el código a código máquina nativo durante el tiempo de publicación. Esto significa que cualquier mecanismo que dependa de la reflexión (inspección o modificación de tipos en tiempo de ejecución) o de la generación dinámica de código (como la mayoría de los ORMs tradicionales) será problemático o directamente incompatible.
    *   **Solución para Dapper AOT:** La elección de Dapper AOT es excelente, ya que está diseñado específicamente para generar el código de mapeo en tiempo de compilación. Las implementaciones de `ISaleRepository` deberán usar `SqlMapper.SourceGenerationContext` y `CommandFlags.None` o `CommandFlags.NoCache` para sus consultas, asegurándose de que todas las operaciones de mapeo de `VentaEntity` y `DetalleVentaEntity` sean resueltas en tiempo de compilación.
    *   **Serialización:** Si la implementación necesita serializar/deserializar datos (ej. para caché o logs), se debe usar `System.Text.Json` con `Source Generation` para garantizar la compatibilidad con AOT.

2.  **Manejo de Consultas N+1 (Detalles de Venta):**
    *   **Riesgo:** Al implementar `GetByIdAsync` o `FindAsync` en un repositorio de ventas, existe un riesgo inherente de generar consultas N+1 al recuperar los `SaleDetail`s asociados a cada `Sale`. Si cada `Sale` se carga con una consulta, y luego cada uno de sus `SaleDetail`s se carga en una consulta separada, esto resultará en N+1 consultas a la base de datos (1 para la venta, N para sus detalles).
    *   **Impacto AOT:** Dapper AOT no resuelve este problema por sí mismo; es una preocupación del patrón de acceso a datos. Aunque AOT optimice la ejecución de cada consulta individual, la explosión del número de consultas sigue siendo un problema de rendimiento.
    *   **Solución:** La implementación debe emplear técnicas como:
        *   **Consultas Multi-Map de Dapper:** Utilizar la funcionalidad `Query<T1, T2, TResult>(...)` de Dapper para cargar `VentaEntity` y `DetalleVentaEntity` en una sola consulta SQL que une ambas tablas, y luego construir el objeto `Sale` y sus `SaleDetail`s en memoria.
        *   **Carga Eager (Eager Loading):** Recuperar todos los detalles de venta relevantes en una consulta separada y luego "coserlos" a las ventas correspondientes en memoria (utilizado en el ejemplo del `SaleMapper`). Esto reduce el número total de viajes a la base de datos.

3.  **IAsyncEnumerable y Streaming de Datos:**
    *   **Beneficio:** La interfaz utiliza `IAsyncEnumerable<Sale>`, lo que es perfecto para AOT. La implementación concreta podrá utilizar `yield return` asíncrono con un `DataReader` de Dapper para devolver productos uno a uno (o en pequeños lotes definidos por el `DataReader` del proveedor de la base de datos), minimizando el uso de memoria sin cargar toda la colección en memoria.
    *   **Consideración:** Es vital que la implementación real del repositorio (ej. `DapperSaleRepository`) *no* materialice toda la colección con `.ToList()` o `.ToArray()` antes de retornar `IAsyncEnumerable`, ya que esto anularía los beneficios de streaming.

4.  **Optimización del Código Generado:**
    *   AOT se beneficia de código lo más estático y predecible posible. Las implementaciones del repositorio deben usar construcciones de C# 14 que no impliquen dinamicidad, como constructores primarios y propiedades auto-implementadas para los DTOs de persistencia. Los mappers (ej. `SaleMapper`) ya están diseñados bajo esta premisa.

En conclusión, la interfaz `ISaleRepository` establece un contrato sólido y AOT-friendly. La clave para su éxito en "utmmarket" radicará en una implementación de infraestructura que respete estas restricciones y aproveche las capacidades de Native AOT y Dapper AOT para ofrecer un rendimiento óptimo y un bajo consumo de recursos, mitigando activamente problemas como las consultas N+1.
